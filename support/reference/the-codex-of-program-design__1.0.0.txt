命名约定
命名不是代码表面的装饰，而是结构与语义的第一层投影。一个稳定系统必须让文件名、目
录名、类型名、变量名与动作名共同构成同一套语言，否则维护者只能不断在上下文中猜测
真正含义。本章给出本书使用的命名体系：它既规范“怎么写”，也约束“为什么这样写”。


1.1 通用代码命名约定
在代码内部，变量、常量、函数与类仍需要遵守语言级命名习惯；在文件与目录层面，还需要
额外的结构分隔规则。这两层并不冲突，而是互补。

                              表 1.1: 标识符大小写约定
类别       约定                              例子
普通变量     camelCase                       activeTabId, userConfig
布尔变量     camelCase + is/has/can/should   isEditable, hasPermission
全局常量     UPPER_SNAKE_CASE                MAX_RETRY_COUNT
类名       PascalCase                      TabManager, UserRepository
函数名      camelCase 动宾结构                  getUserById, calculateTotal


在这一层，大小写约定解决的是语言可读性；但一旦跨入文件系统与模块结构，大小写远远
不够。


1.2 理论基础
主流命名法如 camelCase、PascalCase 与 snake_case，都存在同一个结构性缺陷：它们倾向于
用同一种分隔机制表达不同层级的关系。这样写出来的长名称虽然可读，却无法机械地恢复
原始层次。

本章采用的核心思想是：不同层级，应使用不同分隔符。这让名字既能被人读懂，也能被工
具稳定解析。

    1.4 与大小写体系的比较
    本体系不是否定 camelCase 或 PascalCase，而是指出：大小写适合语言内部标识符；文件系统、
    目录树与跨模块制品需要额外结构线索。以双下划线标记主体，可以显著降低长文件名的语
    义歧义。


    1.5 片段质量标准：S-I-D 原则
    良好的命名片段应满足 S-I-D 三原则：

    Specific 使用足够具体的词，避免 data、item 这类空洞容器词。

    Identifiable 名称必须能被直接搜索、讨论与复述，避免晦涩缩写。

    Differentiated 名称之间必须具有足够差异，避免 userService、userManager、userHandler
    在无语义区分时并存。

    S-I-D 约束的是命名片段的质量，而不是形式。形式正确但语义空洞的名字，同样是坏名字。


    1.6 A/HC/LC 模式映射
    结构上可以把名称拆成三层：A 表示属性集合，HC 表示高层语境，LC 表示局部限定。是否
    写成完整三层，应由歧义程度决定，而不是机械补全。

    后缀不应滥用。只有当后缀真正承担类型区分作用时，才值得保留，例如 Manager、Registry、

层级不是越多越好。名称应在消除歧义所需的最小复杂度上停止。最常见错误包括：重复目
录上下文、使用语义空洞的泛词、滥用后缀、在同一项目中混用多套动作词，以及用缩写掩
盖真正语义。

命名是一种架构工具。一个系统若连最小单位的名字都无法稳定表达，其更高层的组织几乎
不可能长期稳定。

通用语言 与 领域建模
命名规则解决“怎么写”，通用语言解决“到底该叫它什么”。一旦一个系统在核心概念上没

2.1 命名不一致的成本

字判断真实边界。命名不一致不仅增加沟通成本，也直接破坏结构判断、搜索定位与接口设
计。


2.2 起点：DDD 中的通用语言
领域驱动设计给出的关键启示是：一个概念，在一个边界内应当只有一个名字；一个名字，也

5.1 概览与分类

为什么需要设计模式

设计模式的价值在于把反复出现的问题及其解法显式化。对大型系统而言，它至少降低三类
成本：沟通成本、维护者的心理预期成本，以及面对新问题时“从零发明结构”的成本。但模
式并非免费午餐，每个模式都会引入额外层次、更多对象或更远的调用路径。

6.1 设计目标与总原则
程序架构的目的，不是造出一棵“看起来漂亮”的目录树，而是让运行时所有权清晰、依赖
方向可理解，并阻止代码库滑向循环引用、壳层文件承载业务逻辑、或初始化路径四处散落
的状态。

架构首先是一张所有权地图

一个合格的项目结构，应能快速回答以下工程问题：

- 运行时从哪里开始启动

- 系统级能力由谁持有

- 最小可复用 UI 粒子放在哪里

- 窗口壳层由谁定义

- 新的业务行为可以在何处接入而不污染基础设施

如果结构回答不了这些问题，它就不是架构，只是装饰。

物理结构必须服从运行时职责

代码库的物理布局，应当投影出系统真实的运行时角色。在本项目中，核心关注点是可复用
UI 粒子、可执行能力模块，以及壳层级宿主表面。因此，结构必须先把这些关注点拆开，才
允许局部实现细节逐渐积累。

当前术语优先于历史原型标签

历史路径里也许还残留旧标签，但正式文档应采用 第 2 章 已经建立的当前词汇。因此，本章
以 Element、Component、Page、Layout、Container 为视图系统的权威语言，而不是沿用早期

7.1 设计目标与总原则
交互层存在的目的，是把“触发形式”与“执行逻辑”解耦。键盘快捷键、按钮点击、菜单选
择、触控手势，都是异质触发源；系统必须把它们统一翻译成命令导向的执行路径。

无逻辑 UI

UI 控件不得承载业务逻辑。诸如 sys-btn、menu-item、toggle 等视图层元素，只是发射器。
它们的职责应当收缩为暴露或发射一个 Command ID。

输入多态

不同输入来源可以表达同一用户意图，因此内核应识别 Command ID，而不是依赖具体的物理
触发源。

原子能力

每 一 项 可 执 行 用 户 能 力 都 应 表 示 为 一 个 原 子 命 令 单 元， 例 如 file:save、edit:copy、
view:toggle-sidebar。命令是交互层中最小、可独立寻址的执行单位。


显式解耦

交互层必须显式分离以下关注点：

- 输入捕获

- 输入到命令的翻译

- 命令查找与派发

- 业务执行

执行数据流

命令是无状态执行单元，运行时数据应通过 context 注入。有效的数据流是：UI 触发 →
Command ID → Registry.resolve(id) → Command.execute(context) → service 执行。

这样的设计使日志记录、排队、宏录制以及未来扩展执行模型都更一致。

Command Context 是执行封套

command context 不是随手塞参数的袋子，而是一次命令调用的执行封套。命令之所以“无状
态”，不是因为什么都不知道，而是因为与本次调用有关的运行时事实都从外部通过 context
提供。

交互层应在派发时组装这个封套；命令读取它、完成意图，然后返回，不保留可变会话状态。

上下文边界规则

为了保持有用性，command context 应遵守以下规则：

1. 延后组装。在派发时创建 context，以反映最新作用域、选择与环境。

2. 只携带必要信息。只包含当前命令所需的最小充分事实。

3. 不要把 context 当持久状态。长寿命工作流数据、编辑历史与可恢复 UI 状态不属于命令封

4. 优先传递抽象而非原始 UI 对象。能传 service facade、已选实体或 editor adapter 时，就不应
  传大而可变的视图对象。

5. 命令外部拥有上下文。交互层负责构造，命令只消费，不得私藏为会话状态。

状态管理 与 核心服务
本章是 第 7 章 的工程续篇。第 7 章定义用户意图如何经由命令、快捷键与发射器进入系统；
本章则定义这些命令执行之后，系统如何保存、恢复、校验并迁移运行时状态。重点不在“做
一个大 Store”，而在服务层所有权、文件生命周期、上下文恢复、配置分层与系统反馈的治理。


8.1 设计目标与总原则
状态管理层的目的，不是把一切变化中的值都塞进某个全局容器，而是确保运行时状态拥有
清晰拥有者、合法生命周期，以及可恢复的表示形式。

逻辑必须脱离具体 UI

业务服务不应依赖具体控件。文件 I/O、配置管理、视图状态持久化与反馈编排属于逻辑层，
而不是按钮、页面或编辑器组件。

切换必须视为事务

从文件 A 切到文件 B，再返回文件 A，并不是简单地改一下指针，而是由状态捕获、源切换
与状态恢复组成的一次事务。只做中间步骤的设计，最终一定会丢失用户上下文。

状态不是倾倒场

系统应仅保存保证正确性与恢复性所必需的状态。派生值在条件允许时应重算，而不是盲目
缓存。

所有权必须显式

每一类重要状态都必须有清晰拥有者：

- 文件生命周期状态属于 file service

- 视图恢复快照属于 context state manager

- 偏好值属于 configuration service

- 瞬时系统通知属于 feedback service

一旦所有权含混，调试与恢复都会失去可靠性。

     8.2 服务内核

     服务是运行时单元

     主要逻辑能力应表现为由应用内核管理的 service。service 是一个长生命周期运行时单元，具
     有显式的初始化、释放与变更通知职责。

                                          代码 8.1: 基础服务契约

          BaseService

 1    class BaseService extends EventTarget {
 2          async init() {}
 3

 5

     这个基础契约带来三项保证：

     1. 内核可以按已知顺序初始化服务

     2. 内核可以在关闭或重启时可预测地释放服务

     3. 依赖层可以观察有意义的状态变化，而不产生直接耦合

     为什么服务必须由内核拥有

     如果 service 由视图随意实例化，很快就会出现三类故障：生命周期碎裂、状态所有权重复、

     本地文件、数据库文档与云端内容都能通过同一逻辑 API 被处理。

     这一抽象的价值不在“纯洁”，而在于存储拓扑改变时，编辑器层不需要跟着一起改写。

8.5 配置服务

配置是一组经过校验的数据

配置值不是杂项键值对，而是产品契约的一部分，因此必须具备校验、优先级规则与可预测
运行行为。

核心职责包括：

- schema 校验

- 在安全范围内支持热更新

- 跨作用域持久化

- 向依赖子系统广播变更

分层配置优先级

一个实用的设置系统至少应支持如下优先级：


     default < user < workspace


这使产品能够提供安全默认值、尊重用户偏好，并在协作场景下允许项目级覆盖。

并发控制 与 资源管理
本章是 第 7 章 与 第 8 章 的运行时续篇。第 7 章解释用户意图如何被归一化为命令，第 8 章

9.1 设计目标与总原则
桌面应用中的并发控制，不是为了让所有事情都尽可能同时发生，而是为了避免对特权资源
的冲突访问，使所有权边界保持显式，并确保窗口、对话框与 IPC 路由在重复触发或重叠触
发时仍然具有确定性。

特权能力必须有单一拥有者

窗口创建、原生对话框、应用生命周期钩子与特权 IPC 处理器都应由主进程拥有。这些能力
不是普通 UI 关切，不能被当作 renderer 本地行为处理。

进程边界就是架构边界

renderer 并不是 main 的轻量延伸，而是拥有不同信任级、不同生命周期与不同故障模式的独
立运行边界。因此，架构必须显式表达进程所有权。

并发控制是选择性的，不是全局性的

并非所有动作都需要串行化。并发控制应重点落在独占、昂贵、安全敏感或经由操作系统中
介的资源上，例如原生对话框、窗口生成、关闭协调与特权 IPC 入口。

重复触发必须无害

快速连续输入不应制造重复窗口、重复对话框或重叠破坏性操作。如果同一意图在短时间内
多次到达，系统必须根据策略选择聚焦、合并、忽略或串行。

主进程职责

主进程应拥有应用启动与关闭、窗口创建与销毁、原生对话框与 OS 集成、特权 IPC 路由，以
及进程级故障的恢复编排。

渲染进程职责

renderer 应集中于视图渲染、局部交互与命令发射。它可以请求特权工作，但不得自行决定窗
口权威，也不得绕开进程边界。

辅助进程是安全阀

如果某任务计算量大、易崩溃或具备独立操作意义，就应尽量移出 main。并发控制不仅是
“加锁”，也是避免把一切高风险工作塞进同一个事件循环。

窗口注册表是真相来源

多窗口应用需要显式窗口注册表。这个注册表既要按运行时身份识别窗口，也要按语义角色
识别窗口。

没有注册表，重复触发很快就会退化成重复窗口与泄漏引用。

为什么原生对话框需要串行化

原生对话框属于 OS 中介下的独占资源。如果 renderer 可以无协调地连续触发同一个 open-file

这正是一个“小型互斥”能换来巨大稳定性的场景。

    共享通道契约

    IPC 通道名必须作为共享契约被集中定义，而不是散落成各个 renderer 与 main handler 中的字
    符串字面量。

    代表性例子包括：

    - window:minimize

    - window:open-settings

    - dialog:open-file

    - app:request-quit

    这样做能让路由可检查，并减少通道语义漂移。

    主进程集中路由

    主进程应集中注册 IPC handler，以便权限检查、sender 校验与生命周期策略能够在同一处被
    看见。

                            代码 9.2: 单例辅助窗口的主进程 IPC 路由

       ipcMain.handle

1    ipcMain.handle('window:open−settings', async (event) => {
2          if (!validateSender(event.senderFrame)) return null;
3          return windowManager.createOrFocus('settings');
4    });




    sender 校验是强制项

    IPC 不是因为来源于“应用内部”就天然可信。只要特权工作通过 IPC 暴露，就必须在执行动
    作或返回敏感数据之前校验 sender。这既是安全要求，也是并发要求：未经授权或重复来源
    不应争夺主进程特权能力。

    广播与请求-响应必须区分

    IPC 至少应区分两种完全不同的通信形态：

    - 带校验、带确定结果的 request-response，用于特权操作

    - 面向非特权状态更新的 broadcast / event fan-out

    把这两类模式混入一个模糊 event bus，只会让竞态更难分析。

9.6 生命周期协调与故障围堵

单实例协调

如果应用在概念上是单实例，就应显式协调第二次启动。通常的正确行为不是允许多个主实
例竞争同一套应用级资源，而是：启动时请求单实例锁；如果失败，则把意图转发给主实例
并退出；在主实例里恢复并聚焦相关窗口。

优雅关闭是一笔事务

应用关闭应被视为协同事务，而非原始进程退出。

                       表 9.4: 优雅关闭的阶段
阶段            职责
Validation    询问脏状态或挂起工作是否阻止关闭
Persistence   允许 renderer 持有的状态先持久化，或在产品要求下否决关闭
Teardown      释放窗口级与进程级资源，并注销全局钩子
Release       清空注册表、释放锁，仅在清理完成后允许最终退出


绕过这些阶段的“立即退出”，只能保留给极端异常情况。

崩溃可见性与恢复边界

当 renderer 或 child process 崩溃时，应用应在对应拥有边界处响应，而不是默默污染全局假设。
最低要求包括：记录故障事件、废弃失效进程或窗口的旧引用、在用户仍有选择权时暴露恢
复方案，以及仅通过正常 manager 路径重建表面。

10.1 设计目标与总原则
工程文档中的算法章节，不是为了复述一本通用算法教材，而是为了识别本系统中反复出现、
真正影响运行时表现的计算问题，并为这些问题设定稳定、可预测的处理规则。

算法是运行时契约

在本系统中，算法不是实现层的小技巧，而是运行时契约。作用域解析决定物理输入最终命
中哪个命令；优先级调度决定严重错误会不会被低价值提示淹没；依赖排序决定服务与插件
能否无死锁启动；增量更新决定拖拽、搜索与布局调整能否保持交互性，而不是退化为反复
的全量重算。

先保证正确性，再谈微优化

一个常数项更小但结果错误的算法，不是优化，而是缺陷。工程顺序应当始终是：先得到正
确结果，明确边界行为，再控制复杂度落在可接受范围内，最后才考虑常数级提速。

复杂度必须服从交互预算

同一个复杂度在不同运行路径上的工程意义并不相同。启动阶段的一次 O(n) 扫描可能合理，
但把同样的扫描放到每次按键、拖拽帧或滚动事件里，就可能直接破坏体验。因此，复杂度
讨论必须始终绑定到用户感知延迟与触发频率。

局部变化优先增量更新

当系统已经处于有效状态，且变化只影响一个局部切片时，优先策略应当是更新受影响部分，
而不是重算整棵树、整份文档或整个工作台窗口。这一原则同时适用于布局、搜索索引、状
态恢复与反馈渲染。

10.5 搜索、过滤与稳定排序

推荐的计算流水线

面向用户的搜索界面，应采用稳定的处理流水线：

1. 归一化原始输入与候选文本

2. 过滤候选集

3. 对剩余候选打分

4. 使用显式的并列打破规则进行排序

5. 只在排序完成后再裁剪或分页

这个顺序很重要。先排序后归一化，会放大大小写与别名噪声；先打分后过滤，则会浪费算
力在本就应被排除的候选项上。

排序必须稳定

用户看到的排名不仅需要主分数，还需要可预测的并列打破规则。当多个候选分数相同，运
行时应当回退到第二语义键，最后再回退到稳定身份键。否则同一个查询可能连续几次得到
抖动的结果顺序，逐步损耗用户信任。

重复搜索应依赖索引

如果同类查询会反复出现，系统就不应每次都重新清洗原始数据并做全量扫描。归一化关键

公式更有价值。


10.6 优先级调度、队列策略与有界历史

为什么 FIFO 常常不够

反馈系统是最直观的例子。严重错误不应排在一串低价值成功提示之后。普通先进先出不能
满足这一要求，必须有显式的优先级感知调度。

抢占式优先级队列

抢占式优先级队列能提供三种关键性质：高优先级项可以中断低优先级展示；同优先级项仍
保留可解释的顺序；当前显示与历史保留可以分开管理。当“紧急程度”和“可见性”比单
纯到达顺序更重要时，这是更合适的模型。

为什么需要拓扑顺序

一旦系统里出现“A 必须先于 B 加载”这样的规则，问题就不再是普通列表排序，而是有向
无环图上的依赖排序。服务启动、插件装配与按依赖组装模块，都会落入这一类。

循环检测是必需项

拓扑排序只在无环条件下成立。因此，一个不检测环的依赖装配系统是不完整的。出现循环
时，运行时应当显式失败、指出循环路径，并停止启动，而不是把结构错误放任成随机的部
分初始化。

启动顺序应被声明为数据

当启动已经成为一个拓扑问题，就不应再把顺序藏在冗长的 if-else 或脆弱的手工 import 顺序
里。更合理的设计是把依赖关系声明成数据，再由一次拓扑遍历计算最终执行顺序。


10.8 增量计算与局部重建

脏标记与版本戳

当系统只改动了一小部分，与其全量重扫，不如标记发生变化的片段、使相关缓存失效，并
仅重算受影响的区域。脏标记与版本戳虽然简单，却是高价值的增量工具。

比例状态往往比绝对值更稳定

若布局本质上依赖容器尺寸，保存绝对像素值通常会在后续 resize 或重新装配时制造不必要
的重算压力。比例状态、相对偏移和标准化的父子关系，在恢复时通常更稳健。

局部重建优于全局失效

隐藏一个面板，不应要求整个工作台全量重建；修改一个节点，也不应触发与其无关的全局
重排序。当所有权边界清晰时，优先策略应当是局部差异计算、局部重排、局部快照恢复与
局部索引失效，而不是系统级重置。


10.9 算法策略需要升级的信号
以下症状通常说明系统需要的是更强的算法策略，而不是继续堆条件分支：

1. 同一类查询形态下，数据被反复完整重扫

2. 相同用户动作随着数据量增长出现明显延迟分化

3. 等价查询的结果顺序持续抖动

4. 启动顺序依赖人工维护，并会因小改动失效

11.1 版本号混乱的代价
常见场景是：用户报告“Beta 3 有 bug”，但团队内部已经说不清那是三周前哪个提交、哪一
次构建、哪一条分支。根本问题在于：版本号本该是一种自描述信息载体，但大多数项目把
它只当作递增计数器。

版本号混乱通常表现为三种退化：

1. 数字膨胀。一周三次构建，PATCH 从 1 跳到 47，但没有人知道每次递增背后发生了什么。

2. 阶段标签失真。 Alpha 被发给真实用户，Beta 被发给 QA，标签与受众错位，导致整个版
 本体系失去可信度。

3. 构建不可追溯。手里只有一个 app-setup.exe，却说不清它对应哪条分支、哪个提交、哪
 一次流水线构建。

完整的版本与发布系统，解决的不是“怎么给一个数字”，而是：给定一个版本字符串，开发、
测试、用户与 CI 都能独立推断该版本的稳定性承诺、功能边界、目标受众与所处环境。


11.2 基础：Semantic Versioning 2.0.0
Semantic Versioning 2.0.0（SemVer）是当前行业最广泛采用的版本格式规范。本系统以 SemVer
作为格式基础，再叠加 第 2 章 定义的生命周期语义。

核心规则

本系统直接采纳三条铁律：

1. 发布后不可变。一旦 1.2.0 发布，其二进制内容不可被静默替换，任何修复都必须发布为
 新版本。

2. 递增即契约。 MINOR 递增时 PATCH 归零；MAJOR 递增时 MINOR 与 PATCH 同时归零。
